LLM Gateway maintains a catalog of available models with metadata about capabilities, pricing, failover behavior, and provider configuration.

## Endpoints

```
GET    v1/models                # List all models
POST   v1/models                # Register a model
GET    v1/models/:model_id      # Get model details
PATCH  v1/models/:model_id      # Update model (partial)
DELETE v1/models/:model_id      # Remove model
```

### Fields

| Field | Type | Description |
|-------|------|-------------|
| `model_id` | string | Unique model identifier (used in API calls) |
| `type` | string | Model type: `completion`, `embeddings`, or `image_generation` |
| `display` | object | User-facing display info (name, brand, icon, subtitle, description) |
| `capabilities` | object | What the model can do (`text`, `vision`, `image`, `audio`, `file`) |
| `pricing` | object | Cost per 1M tokens (input/output) |
| `limits` | object | Context window (`context`), max output (`response_tokens`), rate limits (`rate.rpm`, `rate.tpm`) |
| `failover` | string | Model to fall back to on failure (note: failover is currently driven by governance config, not this field) |
| `region` | string | Deployment region (used for carbon footprint emission factors) |
| `metrics` | object | Eco and cost metrics (`energy_per_token`, `pue_profile`, `eco_score`, `cost_tier`, `training_date`) |
| `provider_config` | object | Provider-specific config (batch sizes for embeddings, prompt cache, additional body fields) |
| `tags` | array | Filterable tags for routing and filtering |
| `enabled` | boolean | Whether the model is available |

## Model Routing

LLM Gateway supports intelligent model routing — selecting the best model for a request based on configurable strategies.

### Routing Strategies

| Strategy | Description |
|----------|-------------|
| `rules` | Rule-based routing (if/then logic based on request properties) |
| `llm_classifier` | Use an LLM to classify the request and pick the best model |
| `capabilities` | Match request requirements to model capabilities |
| `cost_optimized` | Pick the cheapest model that meets requirements |
| `hybrid` | Combine multiple strategies with weights |

The `_route-model` automation implements routing logic:

```yaml
# Simplified routing flow
1. Parse request requirements (tools needed? vision? json mode?)
2. Filter models by capabilities
3. Apply strategy:
   - rules: evaluate rule chain
   - capabilities: score models by capability match
   - cost_optimized: sort by price, pick cheapest
4. Return selected model
```

### Example: Rule-Based Routing

```yaml
routing:
  strategy: rules
  rules:
    - condition: "tools in request"
      model: gpt-4o
    - condition: "message length > 50000 tokens"
      model: gpt-4o-mini
    - default: gpt-4o
```

## Provider Resolution

When a model is selected for chat completions, `_resolve-provider` finds the connection config by scanning each provider's `models` list in order:

```
openai → azure_openai (prismeai set) → openailike → anthropic → fallback to openai
```

The automation does **not** look up a `provider` field on the model — it iterates through provider configs in `index.yml` until it finds a match. If no provider claims the model, it falls back to OpenAI.

Provider-specific parameter exclusions (e.g., `top_p` for Anthropic, `max_tokens` for some Azure deployments) are applied automatically.

> **Note**: Vertex and Bedrock models are not resolved by `_resolve-provider`. They are handled by the **OpenAI app import** (used for embeddings and image generation), not by the direct HTTP calls used for chat completions.

## Model Aliases

Some providers use non-standard model identifiers. The `model_aliases` config maps friendly names to provider-specific IDs:

```yaml
model_aliases:
  gemini-pro: 'projects/my-project/locations/us-central1/publishers/google/models/gemini-pro'
  claude-3: 'anthropic.claude-3-sonnet-20240229-v1:0'
```

## Data Storage

Models are stored in the `models` collection (via Collection app):

| Field | Type | Indexed |
|-------|------|---------|
| `model_id` | text | Yes (unique) |
| `type` | text | Yes |
| `enabled` | boolean | Yes |
| `tags` | json | Yes |
| `display` | json | — |
| `capabilities` | json | — |
| `pricing` | json | — |
| `limits` | json | — |
| `metrics` | json | — |
| `provider_config` | json | — |
| `failover` | text | — |
| `region` | text | — |

Model specs are cached in `global.models_cache` (keyed by `model_id`, invalidated on PATCH/DELETE) for carbon footprint computation and embedding batch configuration.
