## Workspace Integration Tests

Each workspace includes integration test suites as private endpoint automations. They exercise the full API stack — authentication, data access, and cross-workspace calls.

### Convention

Test files follow the `tests-*.yml` pattern and are private endpoints:

```yaml
# automations/tests-items-crud.yml
slug: tests-items-crud
name: Tests - Items CRUD
when:
  endpoint: tests/items-crud
private: true
do:
  - comment: Create item
  - fetch:
      url: '{{global.apiUrl}}/workspaces/slug:my-workspace/webhooks/v1/items'
      method: POST
      headers:
        Authorization: 'Bearer {{_wsJwt.jwt}}'
      body:
        name: 'Test Item {{timestamp}}'
      output: createResult

  - comment: Verify creation
  - conditions:
      '{{createResult.id}}':
        __empty__:
          - output:
              status: FAIL
              test: create
              error: No ID returned
          - break:

  - comment: Get item
  - fetch:
      url: '{{global.apiUrl}}/workspaces/slug:my-workspace/webhooks/v1/items/{{createResult.id}}'
      method: GET
      headers:
        Authorization: 'Bearer {{_wsJwt.jwt}}'
      output: getResult

  - comment: Verify get
  - conditions:
      '{{getResult.name}}':
        'Test Item {{timestamp}}':
          - set:
              testsPassed: '{{testsPassed + 1}}'
        __default__:
          - output:
              status: FAIL
              test: get
              error: 'Name mismatch: {{getResult.name}}'
          - break:

  - comment: Cleanup - delete item
  - fetch:
      url: '{{global.apiUrl}}/workspaces/slug:my-workspace/webhooks/v1/items/{{createResult.id}}'
      method: DELETE
      headers:
        Authorization: 'Bearer {{_wsJwt.jwt}}'

  - output:
      status: PASS
      tests_passed: '{{testsPassed}}'
```

### Running Tests

Tests are triggered via HTTP:

```bash
curl -X POST \
  https://api.prisme.ai/v2/workspaces/slug:my-workspace/webhooks/tests/items-crud \
  -H "Authorization: Bearer $TOKEN"
```

### Aggregate Test Runner

Some workspaces have a `tests-run-all` or `tests-all` automation that runs all test suites sequentially and returns an aggregate result:

```yaml
slug: tests-run-all
when:
  endpoint: tests/run-all
private: true
do:
  - tests-items-crud:
      output: test1
  - tests-items-search:
      output: test2
  - output:
      results:
        items_crud: '{{test1.status}}'
        items_search: '{{test2.status}}'
      all_passed: '{{test1.status == "PASS" && test2.status == "PASS"}}'
```

## Frontend Tests

The platform uses **Vitest** for frontend unit and component tests:

```bash
# Run tests
npm run test --prefix services/platform

# Watch mode
npm run test:watch --prefix services/platform
```

Test files follow the `*.test.ts` / `*.test.tsx` convention and live alongside their source files.

## Type Checking

```bash
npm run typecheck --prefix services/platform
```

Runs `tsc --noEmit` to check TypeScript types without generating output files.

## Linting

```bash
npm run lint --prefix services/platform
```

Runs ESLint across the codebase.

## CI Validation

For a full validation pass:

```bash
npm run typecheck --prefix services/platform && \
npm run test --prefix services/platform && \
npm run lint --prefix services/platform && \
npm run build --prefix services/platform
```
