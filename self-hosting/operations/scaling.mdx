## When to Scale

**Performance indicators:**
- API response time P95 > 500ms
- Automation throttling duration > 0
- Memory usage > 85% on any node

**Business indicators:**
- Approaching rate limits regularly
- Growing concurrent user count
- Increasing automation volume

## Core Service Scaling

| Component | Scaling Method | Key Consideration |
|-----------|---------------|-------------------|
| API Gateway | Horizontal (replicas) | Stateless, scale freely |
| Runtime | Horizontal (replicas) | Rate limits are per-instance |
| Platform Frontend | Horizontal (replicas) | Stateless nginx, very lightweight |
| Events | Horizontal (replicas) | WebSocket sessions are sticky |
| Workspaces | Horizontal (replicas) | Stateless CRUD |

```yaml
# HPA example
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: prismeai-runtime-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: prismeai-runtime
  minReplicas: 3
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

## Resource Optimization

<Accordion title="Internal API URL">
  Set `INTERNAL_API_URL` to the cluster-internal address to avoid external network hops:
  ```bash
  INTERNAL_API_URL=http://prismeai-api-gateway.prisme.svc.cluster.local:3001/v2
  ```
</Accordion>

<Accordion title="API Gateway Timeout">
  For LLM-heavy workloads, increase the gateway timeout:
  ```bash
  FETCH_TIMEOUT=120000  # 120 seconds
  ```
</Accordion>

<Accordion title="Event Size Limits">
  Control Redis memory with event size limits:
  ```bash
  BROKER_EMIT_MAXLEN=100000                        # 100KB per event
  BROKER_EMIT_EXECUTED_AUTOMATION_MAXLEN=10000      # 10KB for execution events
  ```
</Accordion>

## Database Scaling

### Elasticsearch/OpenSearch

- Add data nodes for storage capacity
- Add master nodes for cluster stability (3 dedicated recommended)
- Configure ILM policies for automatic index lifecycle management
- Tune: `number_of_shards`, `number_of_replicas`, `refresh_interval`

### MongoDB

- Scale vertically (more RAM) for working set that exceeds memory
- Add secondaries for read scaling
- Use sharding for datasets exceeding single-node capacity

### Redis

- Cluster mode for datasets exceeding single-node memory
- Separate instances for different workloads (cache vs. broker vs. sessions)

## Events Cleanup

Configure automated event cleanup via Helm:

```yaml
prismeai-events:
  retention:
    enabled: true
    workspaceMaxEvents: 100000
    workspaceInactivityDays: 90
    automationExecutedExpiration: 7d
```

This creates a CronJob that periodically cleans old events to prevent unbounded index growth.
