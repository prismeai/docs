Two vector provider workspaces implement the same interface, making them interchangeable backends for the [Storage](/services/storage/overview) service's RAG pipeline.

## Providers

| Provider | Workspace | App Slug | Engine |
|----------|-----------|----------|--------|
| **OpenSearch** | `vector-opensearch` | `VectorOpensearch` | k-NN plugin (HNSW via Lucene) |
| **Elasticsearch** | `vector-elasticsearch` | `VectorElasticsearch` | Native `dense_vector` + `knn` queries |

## Shared Interface

All providers implement these 5 operations:

### create-index

Creates a vector index for storing embeddings.

| Argument | Type | Description |
|----------|------|-------------|
| `index` | string | Index name |
| `dimensions` | integer | Vector dimensions (e.g., 1536) |
| `similarity` | string | Similarity metric (`cosine`, `dot_product`, `l2`) |

Both providers accept normalized names (`cosine`, `dot_product`, `l2_norm`). OpenSearch maps these to its native equivalents (`cosinesimil`, `innerproduct`, `l2`).

Returns: `{ success: true, index: "..." }`

### upsert

Inserts or updates vectors.

| Argument | Type | Description |
|----------|------|-------------|
| `index` | string | Index name |
| `vectors` | array | `[{ id, vector, metadata }]` |

Returns: `{ success: true, count: N, errors: [] }`

### search

Performs similarity search.

| Argument | Type | Description |
|----------|------|-------------|
| `index` | string | Index name |
| `vector` | array | Query embedding |
| `limit` | integer | Max results |
| `filter` | object | Metadata filter (Elasticsearch) |
| `filter_json` | string | Metadata filter as JSON string (OpenSearch — preserves flat dotted keys) |
| `min_score` | float | Minimum similarity score |

Returns: `{ results: [{ id, score, metadata }], total: N }`

### delete-vectors

Deletes vectors by ID or filter.

| Argument | Type | Description |
|----------|------|-------------|
| `index` | string | Index name |
| `ids` | array | Vector IDs to delete |
| `filter` | object | Metadata filter for bulk delete |

Returns: `{ success: true, deleted_count: N }`

### delete-index

Deletes an entire vector index.

| Argument | Type | Description |
|----------|------|-------------|
| `index` | string | Index name |

Returns: `{ success: true }`

## Vector Metadata Schema

All providers store vectors with this metadata:

| Field | Type | Description |
|-------|------|-------------|
| `file_id` | string | Source file identifier |
| `vector_store_id` | string | Parent vector store |
| `chunk_index` | integer | Position in source document |
| `text` | string | Original text chunk |
| `scope` | string | Access scope (`knowledge` or `conversation`) |
| `conversation_id` | string | For conversation-scoped data |
| `expires_at` | datetime | Expiration timestamp |

## Provider Differences

### OpenSearch

- Uses the **k-NN plugin** with `knn_vector` field type and HNSW algorithm (Lucene engine)
- Filters passed as raw JSON string (`filter_json`) because Prisme.ai's template engine interprets dotted keys (e.g., `metadata.scope`) as nested objects. The JSON string workaround preserves flat keys.
- Auth via `Basic` HTTP header from `config.auth_header`
- Does **not** support `$or` filter operators

### Elasticsearch

- Uses native **`dense_vector`** field type with `knn` query
- Filters constructed as standard Prisme objects (ES handles dotted keys natively)
- Supports `$or` operators via Elasticsearch's `should` clauses
- Same `Basic` auth pattern

## How Storage Routes to Providers

The Storage workspace's `_vector-router` dispatches to the correct provider based on the vector store's `provider` field:

```yaml
- conditions:
    '{{provider}}':
      opensearch:
        - VectorOpensearch.search: ...
      elasticsearch:
        - VectorElasticsearch.search: ...
```

Changing provider requires:
1. Update the vector store's `provider` field
2. Re-index all files (embeddings are reusable, but the index structure differs)
