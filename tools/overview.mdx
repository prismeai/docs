Tool providers are workspaces that implement specific capabilities consumed by core products. They follow a **pluggable provider pattern** — multiple implementations of the same interface can be swapped without changing consumer code.

## Common Patterns

All tool providers share these architectural patterns:

### App Registration

Tool workspaces are published as apps via `.import.yml`:

```yaml
app:
  slug: VectorOpensearch
  name: Vector Store - OpenSearch
```

Consumer workspaces install them by slug and call operations directly:

```yaml
- VectorOpensearch.search:
    index: '{{index_name}}'
    vector: '{{embedding}}'
```

### Security Model

All tool workspaces share the same base `security.yml`:

| Role | Permissions |
|------|-------------|
| `editor` | Full workspace management |
| `viewer` | Read workspace and pages |
| `workspace` (API key) | Full file CRUD — the service account pattern |

Access control at the tool operation level is handled by the consuming workspace, not the tool itself.

### Private Helpers

Common private automations:

| Helper | Found In | Purpose |
|--------|----------|---------|
| `_llm` | Guardrails, Memories, Evaluations | LLM Gateway client via workspace JWT |
| `_auth` | Memories, Evaluations | Authentication and authorization |
| `_check-permission` | Memories | Permission verification |
| `_error` | Evaluations | Error response formatting |

### MCP Protocol

Some tools implement MCP JSON-RPC 2.0 for agent integration:

| Workspace | MCP Endpoint | Protocol |
|-----------|-------------|----------|
| AI Collection | `ai-collection/mcp` | `2025-06-18` |
| Prompt Library | `mcp` | `2025-06-18` |
| Bing Search | `bing-search/mcp` | `2025-06-18` |

### Testing

Each tool workspace includes integration test suites (`tests-*.yml`) as private endpoint automations that exercise the full API stack.
