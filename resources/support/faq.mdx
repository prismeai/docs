## General

<Accordion title="What is Prisme.ai?">
  Prisme.ai is an enterprise AI platform for building, deploying, and managing AI agents. It provides a workspace-based architecture where each product (Agent Factory, AI Governance, AI Collection, etc.) is a self-contained workspace with REST APIs, data models, and security rules.
</Accordion>

<Accordion title="What LLM providers are supported?">
  The LLM Gateway supports:
  - OpenAI (GPT-4o, GPT-4o-mini, etc.)
  - Azure OpenAI
  - Anthropic (Claude)
  - Local LLMs via LocalAI or Ollama
  - Any OpenAI-compatible API

  See [LLM Gateway](/services/llm-gateway/overview) for details.
</Accordion>

<Accordion title="Can I run Prisme.ai on-premises?">
  Yes. Prisme.ai is designed for self-hosting on any Kubernetes cluster. All data stays within your infrastructure. See the [Self-Hosting guide](/self-hosting/overview).
</Accordion>

<Accordion title="What databases are supported?">
  - **MongoDB 6.x+** or **PostgreSQL 10+** for structured data
  - **Redis 5.x+** for caching and message brokering
  - **Elasticsearch/OpenSearch 8.x+** for events and search
  - **OpenSearch, Elasticsearch, or Redis Stack** for vector storage
</Accordion>

## Technical

<Accordion title="How does authentication work?">
  Three methods:
  1. **JWT** — OIDC-based sessions for web users
  2. **Access Tokens** — Long-lived tokens for scripts and integrations
  3. **API Keys** — Workspace-scoped keys with RBAC permissions

  See [Authentication](/api-reference/authentication).
</Accordion>

<Accordion title="How do workspaces communicate?">
  Four patterns:
  1. **Cross-workspace fetch** — Service-to-service, caller identified by `run.sourceWorkspaceId`
  2. **API Key + fetch** — Cross-workspace with explicit permissions
  3. **Installed app calls** — Direct automation invocation
  4. **Events** — Async communication via the event bus

  See [Inter-Service Communication](/architecture/inter-service).
</Accordion>

<Accordion title="How are agents different from automations?">
  Agents are built on top of automations. An automation is a single DSUL workflow. An agent adds:
  - LLM-powered reasoning (ReAct loop)
  - Tool dispatch and function calling
  - Memory and context management
  - Multi-turn conversations
  - Configurable profiles (simple → orchestrator)

  See [Agent Factory](/products/agent-factory/overview).
</Accordion>

<Accordion title="What is the Collection App?">
  The Collection App is an ORM layer that turns workspace `imports/*.yml` files into typed database collections with CRUD operations, filtering, sorting, and pagination. It's how product workspaces store their data.

  See [Workspace APIs](/architecture/workspace-apis).
</Accordion>

## Operations

<Accordion title="How do I update the platform?">
  1. Update core services via Helm
  2. Rebuild and deploy the platform frontend Docker image
  3. Push updated workspace configurations

  See [Updates](/self-hosting/operations/updates).
</Accordion>

<Accordion title="How do I backup my data?">
  Back up MongoDB/PostgreSQL, Elasticsearch, Redis, and Kubernetes configuration. See [Backup & Restore](/self-hosting/operations/backup).
</Accordion>

<Accordion title="How do I monitor platform health?">
  - `GET /v2/readiness` — Platform-wide health check (admin only)
  - Workspace test suites — `POST /v2/workspaces/slug:*/webhooks/tests/run-all`
  - AI Governance Observability — Platform metrics and usage analytics
  - Prometheus/Grafana for infrastructure monitoring

  See [Operations — Testing](/self-hosting/operations/testing).
</Accordion>
