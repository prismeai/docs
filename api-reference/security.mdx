Prisme.ai provides multiple layers of security for API interactions, from authentication to network isolation.

## Authentication and Authorization

<Tabs>
  <Tab title="Authentication Methods">
    <CardGroup cols="3">
      <Card title="JWT" icon="key">
        Session-based authentication for web clients. Issued after OIDC or anonymous login.
      </Card>
      <Card title="Access Tokens" icon="ticket">
        Long-lived opaque tokens for integrations and scripts.
      </Card>
      <Card title="API Keys" icon="lock">
        Workspace-scoped keys with fine-grained RBAC permissions.
      </Card>
    </CardGroup>

    See [Authentication](/api-reference/authentication) for implementation details.
  </Tab>
  <Tab title="RBAC Authorization">
    Prisme.ai implements Role-Based Access Control at the workspace level:

    <Accordion title="Roles">
      ```yaml
      authorizations:
        roles:
          editor: {}  # Built-in role

          user:
            auth:
              prismeai: {}  # Assigned to all authenticated users

          workspace:
            auth:
              apiKey: {}  # Used with API keys
      ```
    </Accordion>
    <Accordion title="Rules">
      ```yaml
      rules:
        - role: editor
          action:
            - read
            - update
          subject: workspaces

        - role: editor
          action: manage  # Shorthand for all permissions
          subject: files
      ```

      Each rule specifies:
      - **action** — What can be done (single or list)
      - **subject** — What it can be done to
      - **role** — Which role this applies to (optional — omit to apply to everyone)
      - **conditions** — Restrict to specific cases (optional)
      - **inverted** — Set to `true` for deny rules (optional)
    </Accordion>

    See [Architecture — Authentication](/architecture/authentication) for the full permission model across workspaces.
  </Tab>
</Tabs>

## Network Security

<Steps>
  <Step title="TLS Encryption">
    All API communications use TLS (HTTPS). Self-hosted instances must be configured with valid TLS certificates. Minimum TLS 1.2.

    <Warning>
      Never send sensitive data over unencrypted connections.
    </Warning>
  </Step>
  <Step title="Microservices Isolation">
    The api-gateway is the only publicly exposed service. Backend microservices (Runtime, Workspaces, Events) run in a private network. Internal services trust the `x-prismeai-user-id` header set by the gateway.
  </Step>
  <Step title="IP Restrictions (Self-Hosted)">
    For self-hosted deployments, implement IP restrictions via Kubernetes NetworkPolicy:

    ```yaml
    apiVersion: networking.k8s.io/v1
    kind: NetworkPolicy
    metadata:
      name: api-gateway-policy
      namespace: prisme-system
    spec:
      podSelector:
        matchLabels:
          app: api-gateway
      ingress:
        - from:
            - ipBlock:
                cidr: 10.0.0.0/24
          ports:
            - protocol: TCP
              port: 443
    ```
  </Step>
</Steps>

## JWT Security

<Accordion title="Signing and Rotation">
  JWKs stored in the api-gateway database are automatically rotated:

  | Variable | Description | Default |
  |----------|-------------|---------|
  | `JWKS_ROTATION_DAYS` | Rotation period | 30 days |
  | `JWKS_KTY` | Algorithm family | RSA |
  | `JWKS_ALG` | Signature algorithm | RS256 |
  | `JWKS_SIZE` | Key size | 2048 bits |
  | `ACCESS_TOKENS_MAX_AGE` | JWT expiration | 2592000s (30 days) |

  Public keys: `https://api.your-domain.com/oidc/jwks`

  <Warning>
    If a signing JWK is compromised, delete it from the database and restart api-gateway and runtime services.
  </Warning>
</Accordion>

## Best Practices

<Steps>
  <Step title="Secure Token Handling">
    - Store tokens in secure HTTP-only cookies or secure storage
    - Never expose tokens in URLs or client-side code
    - Implement token refresh mechanisms
    - Revoke tokens when no longer needed
  </Step>
  <Step title="Least Privilege">
    - Use API keys with minimal required permissions
    - Create role-specific tokens for different operations
    - Regularly audit and revoke unused access
  </Step>
  <Step title="Input Validation">
    - Validate data types and formats
    - Sanitize inputs to prevent injection
    - Use schema validation for request bodies
  </Step>
  <Step title="Secure Automations">
    - Use workspace secrets for credentials (`{{secret.NAME}}`)
    - Never store sensitive data in automation code
    - Validate outputs from untrusted sources

    ```yaml
    do:
      - fetch:
          method: POST
          url: '{{secret.API_ENDPOINT}}'
          headers:
            Authorization: 'Bearer {{secret.API_TOKEN}}'
          body:
            data: '{{input.data}}'
    ```
  </Step>
</Steps>

## Self-Hosted Security

<Tabs>
  <Tab title="Infrastructure">
    <CardGroup cols="2">
      <Card title="Kubernetes" icon="shield">
        - Enable Pod Security Policies
        - Implement network policies
        - Use securityContext settings
        - Keep Kubernetes updated
      </Card>
      <Card title="Containers" icon="cube">
        - Use minimal base images
        - Scan for vulnerabilities
        - Don't run as root
      </Card>
      <Card title="Secrets" icon="key">
        - Use Kubernetes secrets or external vault
        - Enable encryption at rest
        - Rotate regularly
      </Card>
      <Card title="Database" icon="database">
        - Enable authentication and encryption
        - Implement network isolation
        - Apply least privilege for DB users
      </Card>
    </CardGroup>
  </Tab>
  <Tab title="SSO Integration">
    <CardGroup cols="2">
      <Card title="OIDC" icon="fingerprint">
        Connect to your OIDC provider (Google, Okta, Auth0, Keycloak):

        ```yaml
        providers:
          MyProvider:
            type: oidc
            config:
              client_id: "your-client-id"
              client_secret: "your-client-secret"
              authorization_endpoint: "..."
              token_endpoint: "..."
              jwks_uri: "..."
              scopes: "openid email profile"
        ```
      </Card>
      <Card title="SAML" icon="id-card">
        Connect to SAML providers (ADFS, Okta, OneLogin, PingIdentity) with attribute mapping and JIT provisioning.
      </Card>
    </CardGroup>

    See [Self-Hosting — Enterprise Authentication](/self-hosting/enterprise/authentication) for full setup.
  </Tab>
</Tabs>
