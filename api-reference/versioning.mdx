Prisme.ai uses URL path-based versioning to ensure backward compatibility while enabling platform evolution.

## Versioning Strategy

<CardGroup cols="2">
  <Card title="URL-Based Versioning" icon="link">
    The version identifier is included in the URL path:

    ```
    https://api.your-domain.com/v2/workspaces
    ```
  </Card>
  <Card title="Major Versions" icon="code-branch">
    New major versions (v1, v2, v3) are released for breaking changes, resource restructuring, or fundamental behavior changes.
  </Card>
  <Card title="Minor Updates" icon="code-commit">
    Within a version, updates add new endpoints, extend existing ones, or fix bugs — always backward compatible.
  </Card>
  <Card title="Version Lifecycle" icon="hourglass-half">
    Each major version goes through: Beta → Stable → Deprecated → Retired.
  </Card>
</CardGroup>

## Current API Version

Version **v2** is the current stable API version with no planned retirement date.

```
https://api.your-domain.com/v2/
```

Key capabilities:
- Workspace management and DSUL storage
- Agent creation and management (via Agent Factory)
- File and knowledge base operations (via Storage)
- Event streaming and WebSocket subscriptions
- Fine-grained RBAC permissions

## Backward Compatibility Policy

Within a major version, Prisme.ai guarantees backward compatibility:

**Will never change:**
- Remove endpoints or fields
- Change the meaning of existing fields
- Add required parameters
- Change error codes in breaking ways

**May change:**
- Add new endpoints
- Add optional fields to responses
- Add optional parameters to requests
- Extend enum values
- Fix bugs that don't change intended behavior

## Date-Based Headers

For fine-grained compatibility, use the `X-Prisme-API-Version` header to pin to a specific date's implementation:

```bash
curl -X GET "https://api.your-domain.com/v2/workspaces" \
     -H "Authorization: Bearer YOUR_TOKEN" \
     -H "X-Prisme-API-Version: 2024-10-15"
```

<Info>
  Date-based versioning is optional. If not specified, you get the latest implementation of the requested major version.
</Info>

## Version Lifecycle

<CardGroup cols="4">
  <Card title="Beta" icon="flask">
    - Early access preview
    - May change without notice
    - Not for production
  </Card>
  <Card title="Stable" icon="circle-check">
    - Fully supported
    - Backward compatible only
    - Recommended for production
  </Card>
  <Card title="Deprecated" icon="triangle-exclamation">
    - Still functional
    - Security patches only
    - Migration recommended
  </Card>
  <Card title="Retired" icon="box-archive">
    - No longer available
    - Returns 410 Gone
  </Card>
</CardGroup>

## Deprecation Process

<Steps>
  <Step title="Announcement">
    Public announcement at least 12 months before retirement, with email notification to active users and deprecation notices in API responses.
  </Step>
  <Step title="Sunset Period">
    Version continues to function with security patches only. Warning headers are included:

    ```
    X-Prisme-API-Deprecation: This API version is deprecated. Please migrate to v2.
    ```
  </Step>
  <Step title="Retirement">
    Endpoints return `410 Gone`. Documentation moves to archive.
  </Step>
</Steps>

## Best Practices

<CardGroup cols="2">
  <Card title="Pin Your Version" icon="code">
    Always explicitly specify the API version:

    ```javascript
    const API_VERSION = 'v2';
    const BASE_URL = `https://api.your-domain.com/${API_VERSION}`;
    ```
  </Card>
  <Card title="Monitor Deprecation Headers" icon="bell">
    Check for version-related response headers:

    ```javascript
    const notice = response.headers.get('X-Prisme-API-Deprecation');
    if (notice) console.warn(`API Warning: ${notice}`);
    ```
  </Card>
</CardGroup>
