The Prisme.ai API supports three authentication methods, each suited to different use cases.

## Authentication Methods

<Tabs>
  <Tab title="JWT Authentication">
    JSON Web Tokens are the primary method for web clients and interactive sessions.

    <Steps>
      <Step title="Obtain a JWT">
        JWTs are issued in two scenarios:

        1. **OIDC authentication** — After authenticating with the api-gateway's OIDC server, clients receive an authorization code exchangeable for a JWT. Find your current JWT in the `access-token` cookie after opening any Prisme.ai page.
        2. **Anonymous authentication** — The `/v2/login/anonymous` endpoint initiates unauthenticated sessions.

        ```bash
        curl -X POST "https://api.your-domain.com/v2/login/anonymous" \
             -H "Content-Type: application/json"
        ```

        Response:

        ```json
        {
          "userId": "anon-123456",
          "sessionId": "session-789012",
          "token": "eyJhbGciOiJSUzI1NiIs..."
        }
        ```
      </Step>
      <Step title="Use the JWT">
        Include it in the Authorization header:

        ```bash
        curl -X GET "https://api.your-domain.com/v2/workspaces" \
             -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..."
        ```
      </Step>
    </Steps>
    <Warning>
      JWTs expire after `ACCESS_TOKENS_MAX_AGE` (default: 30 days). Your application should handle token refresh or re-authentication.
    </Warning>
  </Tab>
  <Tab title="Access Tokens">
    Opaque tokens for longer-term programmatic access — ideal for scripts, integrations, and backend applications.

    <Steps>
      <Step title="Generate an Access Token">
        Authenticated users generate tokens via `/v2/user/accessTokens`:

        ```bash
        curl -X POST "https://api.your-domain.com/v2/user/accessTokens" \
             -H "Authorization: Bearer YOUR_JWT" \
             -H "Content-Type: application/json" \
             -d '{
                   "name": "My Integration Token",
                   "expiresAt": "2027-01-01T00:00:00Z"
                 }'
        ```

        Response:

        ```json
        {
          "name": "My Integration Token",
          "expiresAt": "2027-01-01T00:00:00Z",
          "userId": "user-id",
          "token": "at:1234abcd5678efgh",
          "_id": "token-id"
        }
        ```
      </Step>
      <Step title="Use the Access Token">
        ```bash
        curl -X GET "https://api.your-domain.com/v2/workspaces" \
             -H "Authorization: Bearer at:1234abcd5678efgh"
        ```
      </Step>
    </Steps>
    <Warning>
      Access tokens cannot be recovered if lost. Store them securely and never expose them in client-side code.
    </Warning>
  </Tab>
  <Tab title="API Keys">
    Scoped to specific workspaces with granular permission control.

    <Steps>
      <Step title="Create an API Key">
        Workspace administrators create API keys with specific RBAC rules:

        ```bash
        curl -X POST "https://api.your-domain.com/v2/workspaces/<WORKSPACE_ID>/security/apikeys" \
             -H "Authorization: Bearer YOUR_JWT" \
             -H "Content-Type: application/json" \
             -d '{
                   "name": "Read-only access",
                   "rules": [
                     { "action": "read", "subject": "workspaces" },
                     { "action": "create", "subject": "files" }
                   ]
                 }'
        ```

        Response:

        ```json
        {
          "apiKey": "<apikey>",
          "name": "Read-only access",
          "rules": [
            {
              "action": ["read"],
              "subject": "workspaces",
              "conditions": { "id": "<WORKSPACE_ID>" }
            },
            {
              "action": ["create"],
              "subject": "files",
              "conditions": { "workspaceId": "<WORKSPACE_ID>" }
            }
          ],
          "subjectId": "<WORKSPACE_ID>",
          "subjectType": "workspaces"
        }
        ```
      </Step>
      <Step title="Use the API Key">
        Include it in the `x-prismeai-api-key` header:

        ```bash
        curl -X GET "https://api.your-domain.com/v2/workspaces/<WORKSPACE_ID>" \
             -H "x-prismeai-api-key: <apikey>"
        ```
      </Step>
    </Steps>
    <Info>
      API keys follow the principle of least privilege — they only have the permissions explicitly granted during creation. See [AI Governance — API Keys](/products/ai-governance/api-keys) for the platform-level API key system.
    </Info>
  </Tab>
</Tabs>

## JWT Details

<Accordion title="JWT Structure and Signing">
  JWTs are signed with the following defaults:

  - **Algorithm**: RS256 (configurable via `JWKS_ALG`)
  - **Key Type**: RSA (configurable via `JWKS_KTY`)
  - **Key Size**: 2048 bits (configurable via `JWKS_SIZE`)

  Decoded JWT payload example:

  ```json
  {
    "prismeaiSessionId": "session-id",
    "jti": "...",
    "sub": "user-id",
    "iat": 1744705420,
    "exp": 1747297420,
    "scope": "events:write events:read webhooks pages:read files:write files:read",
    "client_id": "prismeai-studio-client",
    "iss": "...",
    "aud": "..."
  }
  ```
</Accordion>

<Accordion title="Key Rotation">
  The api-gateway automatically handles JWT key rotation:

  1. JWTs are signed using a JSON Web Key (JWK) stored in the api-gateway database
  2. JWKs are rotated according to `JWKS_ROTATION_DAYS` (default: 30 days)
  3. Rotated JWKs remain available for verifying existing JWTs
  4. Old JWKs are removed after `ACCESS_TOKENS_MAX_AGE` once all signed JWTs should have expired
  5. Events (`gateway.jwks.updated` and `runtime.jwks.updated`) synchronize all instances

  <Warning>
    If a signing JWK is compromised, manually delete it from the database and restart both api-gateway and runtime services.
  </Warning>
</Accordion>

<Accordion title="Public Keys">
  Public keys for verifying JWTs are available at:

  ```
  https://api.your-domain.com/oidc/jwks
  ```

  Returns keys in JWKS (JSON Web Key Set) format.
</Accordion>

## Authentication Flow

1. **Client authenticates** via OIDC or anonymous login to obtain a JWT
2. **API Gateway validates** the token and adds an `x-prismeai-user-id` header
3. **Request is routed** to the appropriate microservice with user context
4. **Authorization check** — the target service checks permissions via RBAC

<Warning>
  Backend microservices rely on the `x-prismeai-user-id` header for identification. This header is overwritten by the api-gateway — do not set it in client requests.
</Warning>

## Configuration

| Variable | Description | Default |
|----------|-------------|---------|
| `JWKS_ROTATION_DAYS` | Key rotation period in days | 30 |
| `JWKS_KTY` | JWK algorithm family | RSA |
| `JWKS_ALG` | JWK signature algorithm | RS256 |
| `JWKS_SIZE` | JWK size in bits | 2048 |
| `ACCESS_TOKENS_MAX_AGE` | JWT expiration in seconds | 2592000 (30 days) |

## Code Examples

<Tabs>
  <Tab title="Node.js">
    ```javascript
    const response = await fetch('https://api.your-domain.com/v2/workspaces', {
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${token}`
      }
    });

    if (response.status === 401) {
      // Token expired — refresh or re-authenticate
    }

    const data = await response.json();
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import requests

    headers = {
        'Content-Type': 'application/json',
        'Authorization': f'Bearer {token}'
    }

    response = requests.get('https://api.your-domain.com/v2/workspaces', headers=headers)
    response.raise_for_status()
    data = response.json()
    ```
  </Tab>
  <Tab title="cURL">
    ```bash
    TOKEN="your-access-token"

    curl -X GET "https://api.your-domain.com/v2/workspaces" \
         -H "Content-Type: application/json" \
         -H "Authorization: Bearer $TOKEN"
    ```
  </Tab>
</Tabs>
