The HubSpot app provides read/write access to the [HubSpot](https://www.hubspot.com) CRM and Marketing Hub through the HubSpot v3 REST API. It can be used either as a Builder app (automations call HubSpot instructions directly) or as a remote MCP server consumed by an AI Knowledge agent — covering contacts, companies, deals, tickets, engagements (notes, tasks, calls, emails, meetings), associations, pipelines, properties, owners, lists, marketing forms, marketing emails, workflows and the File Manager. Supports both static Private App Tokens (PAT) and OAuth 2.0 authorization-code per-user delegation.

## Prerequisites

- A **HubSpot** account with admin access to the target portal
- **Either** a Private App token with the required CRM + Marketing scopes (Settings > Integrations > Private Apps at `https://app.hubspot.com/private-apps/<portal-id>`), **or** an OAuth application registered in a HubSpot **Developer account** at `https://app.hubspot.com/l/developer`
- Base URL (default: `https://api.hubapi.com`)

    ## Configuration

    | Field | Description |
    |-------|-------------|
    | **HubSpot API Base URL** | Base URL of the HubSpot API (default `https://api.hubapi.com`) |
    | **HubSpot Private App Token (PAT)** | HubSpot Private App access token, stored as a workspace secret. Passed as `Authorization: Bearer <token>`. Optional if OAuth2 is configured. |
    | **OAuth2 Client ID** | HubSpot OAuth Application Client ID. Create an OAuth application at `https://app.hubspot.com/l/developer` (Developer account > Create app > Auth). |
    | **OAuth2 Client Secret** | OAuth Application Client Secret (shown once on the OAuth tab), stored as a workspace secret. |
    | **OAuth Callback URL** | Auto-populated on install — paste this value into the OAuth application's *Redirect URI* at HubSpot. |
    | **OAuth Authorize URL** | Default `https://app.hubspot.com/oauth/authorize`. |
    | **OAuth Token URL** | Default `https://api.hubapi.com/oauth/v1/token`. |
    | **OAuth Revoke URL** | HubSpot has no RFC 7009 revocation endpoint — leave empty. `disconnect` still deletes platform-stored tokens. |
    | **OAuth Scopes** | Space-separated scopes (default covers CRM + Marketing read/write — see [HubSpot scopes](https://developers.hubspot.com/docs/api/working-with-oauth#scopes)). |
    | **Refresh Token TTL (seconds)** | How long the refresh token is kept in user-scoped secrets (default `7200`). |
    | **MCP Endpoint** | Auto-populated on install — URL of the MCP endpoint for this instance. |
    | **MCP API Key** | Auto-populated on install — signed key used in the `mcp-api-key` header. Do not modify. |

    <Note>
      `MCP Endpoint` and `MCP API Key` are generated automatically by the `onInstall` flow and are only needed to expose this instance as an MCP server (see the next tab).
    </Note>

    ## DSUL Examples

    ### Create a Contact and Associate It to a Company

    ```yaml
    - Hubspot.createContact:
        properties:
          email: jane.doe@example.com
          firstname: Jane
          lastname: Doe
          lifecyclestage: lead
        output: contact
    - Hubspot.createAssociation:
        fromObjectType: contacts
        fromObjectId: '{{contact.id}}'
        toObjectType: companies
        toObjectId: '{{company_id}}'
        output: assoc
    ```

    ### Open a Deal in a Specific Pipeline Stage

    ```yaml
    - Hubspot.createDeal:
        properties:
          dealname: '{{opportunity_name}}'
          amount: '{{amount_eur}}'
          pipeline: default
          dealstage: appointmentscheduled
          closedate: 2026-09-30
        associations:
          - to:
              id: '{{contact_id}}'
            types:
              - associationCategory: HUBSPOT_DEFINED
                associationTypeId: 3
        output: deal
    ```

    ### Search Contacts Updated in the Last 7 Days

    ```yaml
    - Hubspot.searchContacts:
        filterGroups:
          - filters:
              - propertyName: lastmodifieddate
                operator: GTE
                value: '{% date().setDays(-7).timestamp %}'
        sorts:
          - propertyName: lastmodifieddate
            direction: DESCENDING
        properties:
          - email
          - firstname
          - lastname
          - lifecyclestage
        limit: 50
        output: contacts
    ```

    ### Send a Transactional Email

    ```yaml
    - Hubspot.sendTransactionalEmail:
        emailId: '{{transactional_email_id}}'
        message:
          to: '{{recipient_email}}'
          from: noreply@example.com
        contactProperties:
          firstname: '{{first_name}}'
        customProperties:
          orderNumber: '{{order_number}}'
        output: send
    ```

    ### Enroll a Contact in a Workflow

    ```yaml
    - Hubspot.enrollContactInWorkflow:
        workflowId: '{{workflow_id}}'
        email: '{{contact_email}}'
        output: enrollment
    ```

  <Tab title="Usage as MCP">
    The HubSpot app ships with a built-in MCP server. Each app instance gets its own signed `mcp-api-key` that encodes the workspace ID and a credentials lookup URL — the HubSpot PAT itself is never passed through headers and is resolved server-side from the app configuration. When OAuth2 is configured, each end user signs in with their own HubSpot account and the MCP server uses the per-user access token.

    ## Plug into an Agent Creator capability

    Agents consume MCP servers directly through **Agent Creator** capabilities. This is the preferred way to expose HubSpot to an agent.

    <Steps>
      <Step title="Create or open a workspace">
        From the Prisme.ai console, create a new workspace (or open the one that will host the connector).
      </Step>
      <Step title="Install the HubSpot app">
        Open the workspace **Imports** panel, search for **HubSpot** and install it.
      </Step>
      <Step title="Configure the credentials">
        Open the freshly installed app instance settings and fill in the required fields (see the *Usage as App* tab for the field-by-field reference). Provide a **Private App Token** for tenant-wide access, an **OAuth Client ID + Secret** for per-user delegation, or both.
      </Step>
      <Step title="Copy the MCP endpoint and API key">
        Still on the app instance configuration page, copy the values of **MCP Endpoint** and **MCP API Key** — both are generated automatically on install.
      </Step>
      <Step title="Open Agent Creator">
        Switch to **Agent Creator** and open the agent you want to extend.
      </Step>
      <Step title="Add a capability">
        Add a new capability to the agent:

        - **If a dedicated HubSpot capability exists** — select it and paste the **MCP API Key** into the `mcp-api-key` field. The server URL is already wired.
        - **Otherwise** — select the generic **`custom_mcp`** capability, paste the **MCP Endpoint** into the *Server URL* field, then open the *Headers* field and add an `mcp-api-key` entry whose value is the **MCP API Key** copied earlier:

          ```json
          {
            "mcp-api-key": "your-mcp-api-key"
          }
          ```
      </Step>
      <Step title="Save">
        The agent now has access to every HubSpot tool exposed by the MCP server.
      </Step>
      <Step title="Brief the agent in its system prompt">
        Wiring the capability is not enough — the agent also needs to know the MCP exists and when to reach for it. Add a short paragraph to the agent's system prompt. Copy-pasteable starter:

        ```text
        You have access to the HubSpot MCP server. Use it whenever the user asks something that maps to HubSpot data — listing, searching, reading, creating or updating contacts, companies, deals, tickets, engagements (notes, tasks, calls, emails, meetings), lists, marketing forms, marketing emails or workflows. Always prefer calling an MCP tool over guessing from prior context, and confirm with the user before any destructive action (archive, batchArchive, merge, deleteList, unenrollContact, removeListMembers).
        ```

        Refine the trigger keywords (resource names, business domains, typical user phrasings) so the agent reliably picks up the right intent in your context.
      </Step>
    </Steps>

    ## Legacy MCP Install (Advanced > Tools)

    Use this flow to plug the HubSpot MCP into an AI Knowledge agent that does not yet support the native MCP picker.

    <Steps>
      <Step title="Install the HubSpot app">
        Install and configure the app in the same workspace as your agent (see the *Usage as App* tab). Once configured, `mcpEndpoint` and `mcpApiKey` are auto-populated.
      </Step>
      <Step title="Copy the MCP credentials">
        Open the app instance config and copy the values of **MCP Endpoint** and **MCP API Key**.
      </Step>
      <Step title="Open your AI Knowledge project">
        Navigate to **Advanced > Tools**.
      </Step>
      <Step title="Add an MCP tool">
        Click **Add** and select the **MCP** tab.
      </Step>
      <Step title="Fill in the endpoint">
        Paste the **MCP Endpoint** URL copied from the app instance.
      </Step>
      <Step title="Add the auth header">
        In the **Headers** field, add the signed API key:

        ```json
        {
          "mcp-api-key": "your-mcp-api-key"
        }
        ```
      </Step>
      <Step title="Save">
        The agent can now list and call HubSpot tools through the MCP endpoint.
      </Step>
    </Steps>

    <Note>
      The signed `mcp-api-key` encodes the workspace ID and the `getConfig` webhook URL. The MCP server validates the signature using the central app secret and transparently fetches the HubSpot credentials (PAT or per-user OAuth token) and base URL from the installed app. Credentials are cached per tenant for 10 minutes.
    </Note>

    ## Output Formats

    Every tool accepts an `outputFormat` parameter that controls the MCP response shape:

    - **`verbose`** (default) — human-readable text for LLM consumption
    - **`structured`** — machine-readable JSON in `structuredContent`
    - **`both`** — both text and structured content

    ## Tool Details

    ### contacts

    Single dispatcher for every contact operation. Choose the operation with `action`.

    ```json
    {
      "name": "contacts",
      "arguments": {
        "action": "create",
        "properties": {
          "email": "jane.doe@example.com",
          "firstname": "Jane",
          "lastname": "Doe",
          "lifecyclestage": "lead"
        }
      }
    }
    ```

    | Parameter | Required | Description |
    |-----------|----------|-------------|
    | `action` | Yes | One of `list`, `get`, `create`, `update`, `archive`, `search`, `batchRead`, `batchCreate`, `batchUpdate`, `batchArchive`, `merge` |
    | `contactId` | for `get`/`update`/`archive` | HubSpot contact id |
    | `properties` | for `create`/`update` | Map of property name → value |
    | `filterGroups` / `sorts` / `query` | for `search` | Search payload (OR between groups, AND within a group) |
    | `inputs` | for `batch*` | Array of `{ id }` (or `{ id, properties }` for creates/updates) |
    | `primaryObjectId` / `objectIdToMerge` | for `merge` | Records to keep / merge in |

    ### deals

    ```json
    {
      "name": "deals",
      "arguments": {
        "action": "create",
        "properties": {
          "dealname": "ACME — Annual Subscription",
          "amount": "12000",
          "pipeline": "default",
          "dealstage": "appointmentscheduled",
          "closedate": "2026-09-30"
        },
        "associations": [
          {
            "to": { "id": "<contact_id>" },
            "types": [
              { "associationCategory": "HUBSPOT_DEFINED", "associationTypeId": 3 }
            ]
          }
        ]
      }
    }
    ```

    `action` accepts `list`, `get`, `create`, `update`, `archive`, `search`, `batchRead`, `batchCreate`, `batchUpdate`, `batchArchive`, `merge`. `pipeline` and `dealstage` reference ids returned by the `pipelines` tool.

    ### associations

    ```json
    {
      "name": "associations",
      "arguments": {
        "action": "create",
        "fromObjectType": "contacts",
        "fromObjectId": "<contact_id>",
        "toObjectType": "companies",
        "toObjectId": "<company_id>"
      }
    }
    ```

    | Parameter | Required | Description |
    |-----------|----------|-------------|
    | `action` | Yes | `list`, `create`, `archive` |
    | `fromObjectType` | Yes | Source object type (`contacts`, `companies`, `deals`, `tickets`, `notes`, `tasks`, `meetings`, `calls`, `emails`) |
    | `fromObjectId` | Yes | Source record id |
    | `toObjectType` | Yes | Target object type |
    | `toObjectId` | for `create`/`archive` | Target record id |

    `list` paginates with `after` / `limit`.

    ### lists

    ```json
    {
      "name": "lists",
      "arguments": {
        "action": "create",
        "name": "Q2 leads — France",
        "objectTypeId": "0-1",
        "processingType": "DYNAMIC",
        "filterBranch": {
          "filterBranchType": "OR",
          "filterBranches": [],
          "filters": [
            {
              "filterType": "PROPERTY",
              "property": "country",
              "operation": { "operator": "IS_EQUAL_TO", "value": "France" }
            }
          ]
        }
      }
    }
    ```

    `objectTypeId` is the HubSpot type id (`0-1` = contacts, `0-2` = companies). `processingType` is `MANUAL`, `SNAPSHOT` or `DYNAMIC`. Use `addMembers` / `removeMembers` with `inputs: [{ id: "<recordId>" }, ...]` for `MANUAL` lists.

    ### marketingEmails

    ```json
    {
      "name": "marketingEmails",
      "arguments": {
        "action": "sendTransactional",
        "emailId": "<email_id>",
        "message": {
          "to": "jane.doe@example.com",
          "from": "noreply@example.com"
        },
        "contactProperties": { "firstname": "Jane" },
        "customProperties": { "orderNumber": "ORD-1042" }
      }
    }
    ```

    `action` accepts `list`, `get`, `create`, `update`, `archive`, `sendTransactional`. The `sendTransactional` action requires the email to be of a transactional type and published. `customProperties` maps to merge tags used in the email template.

    ### workflows

    ```json
    {
      "name": "workflows",
      "arguments": {
        "action": "enrollContact",
        "workflowId": "<workflow_id>",
        "email": "jane.doe@example.com"
      }
    }
    ```

    | Parameter | Required | Description |
    |-----------|----------|-------------|
    | `action` | Yes | `list`, `get`, `enrollContact`, `unenrollContact` |
    | `workflowId` | for `get`/`enrollContact`/`unenrollContact` | HubSpot workflow id |
    | `email` | for `enrollContact`/`unenrollContact` | Email address of the contact to enroll |

    ### connect / disconnect

    OAuth session helpers — only meaningful when OAuth2 is configured on the app instance.

    ```json
    {
      "name": "connect",
      "arguments": {}
    }
    ```

    `connect` returns a `connect_url` the end user must click to authorize HubSpot. After completion, every other tool transparently uses the per-user access token. `disconnect` deletes the stored tokens for the current user.

## Error Handling

| HTTP Status | Error | Solution |
|-------------|-------|----------|
| 401 | Unauthorized | Verify the PAT scopes, or re-authorize OAuth via `connect` |
| 403 | Forbidden | The token is valid but the user/app lacks the required scope (e.g. `crm.objects.deals.write`) |
| 404 | Not Found | Verify ids for contact, company, deal, ticket, list, workflow, etc. |
| 409 | Conflict | Duplicate email on contact create, or list name already in use |
| 422 | Validation | Check property names, enum values and association type ids |
| 429 | Rate Limited | Back off — HubSpot enforces per-portal and per-token quotas |

### Common Issues

**"Not configured"** — Neither a PAT nor an OAuth client is set. Generate a PAT in *HubSpot > Settings > Integrations > Private Apps*, or register an OAuth application at `https://app.hubspot.com/l/developer` and paste the Client ID + Secret in the app configuration.

**"Invalid API key" (MCP)** — The `mcp-api-key` header does not match the central app secret. Reinstall the app instance to regenerate a signed key.

**"Credentials lookup failed"** — The MCP endpoint could not reach the `getConfig` webhook of the installed app. Verify that the app instance is still installed in the expected workspace.

**OAuth per-user delegation** — Each end user must click the `connect` tool once (from the AI agent) to authorize; the agent stores a per-user refresh token. HubSpot does not implement RFC 7009 revocation, so `disconnect` deletes platform-stored tokens but does not call HubSpot — users can also revoke access manually in *HubSpot > Account & Billing > Account Defaults > Integrations*.

**Property does not exist** — Custom properties must be created on the relevant object type before they can be set. Use the `properties` tool (`action: create`) with the target `objectType` to create them first.
