The IAM Context endpoint is the bridge between AI Governance and the platform frontend. It provides everything the shell needs to render: navigation, branding, permissions, and organization data.

## Context Endpoint

```
GET v1/iam/context
```

## Navigation Configuration

The sidebar navigation is fully dynamic — defined in the AI Governance workspace configuration and served via the IAM context endpoint.

### Menu Item Structure

| Field | Type | Description |
|-------|------|-------------|
| `key` | string | Unique identifier |
| `label` | string | Display label |
| `icon` | string | Icon name (from curated icon map in the frontend) |
| `path` | string | URL path |
| `badge` | string/null | Badge text (e.g., "New", "Beta") |
| `external` | boolean | Opens in new tab |

### Categories

Menu items are grouped into collapsible categories:

```json
{
  "category": "AI Products",
  "items": [...]
}
```

The frontend renders categories as collapsible sections in the sidebar. Items without a category appear at the top level.

### Dividers

A special item type for visual separation:

```json
{
  "type": "divider"
}
```

## Branding

Organizations can customize the platform appearance:

| Setting | Description |
|---------|-------------|
| `logo` | Organization logo URL (replaces Prisme.ai logo) |
| `colors.primary` | Primary brand color |
| `colors.secondary` | Secondary brand color |
| `fonts.heading` | Heading font family |
| `fonts.body` | Body text font family |
| `custom_css_url` | URL to a custom CSS file for advanced theming |
| `platform_name` | Custom platform name (replaces "Prisme.ai") |

Branding is applied by the `IAMProvider` in the platform shell. CSS variables are set on the root element, and the custom CSS file (if any) is injected as a `<link>` tag.

## Onboarding

When a user has no organization membership, the platform shows an **onboarding modal**:

1. **Join with invite code** — Enter a code shared by an organization admin
2. **Create organization** — Available only to users with `PlatformAdmin` role
3. **Auto-join** — If join rules match the user's email domain, they're automatically added

The onboarding flow is handled entirely by the frontend using the IAM context response.

## Organization Switching

```
POST v1/iam/switch-org
```

```json
{
  "orgSlug": "beta-inc"
}
```

Switches the user's active organization. Updates `user.meta.currentOrg`. The frontend reloads the IAM context to get the new organization's navigation, branding, and permissions.

The organizations list from the IAM context populates the org switcher dropdown in the sidebar.

## Degraded Mode

If no AI Governance workspace is configured (or it's unreachable), the platform shell falls back to **degraded mode**:

- Only the Builder link is shown in the sidebar
- No branding customization
- No organization context
- Basic user info from the OIDC session

This ensures the platform remains usable for development even without the governance layer.

## Service Account Context

```
GET v1/service-accounts/:id/context
```

Service accounts (used for machine-to-machine access) have their own context endpoint that returns permissions and scopes without navigation or branding.
