The IAM Context endpoint is the bridge between AI Governance and the platform frontend. It provides everything the shell needs to render: navigation, branding, permissions, and organization data.

## Context Endpoint

```
GET v1/iam/context
```

This is the first API call the platform shell makes after authentication. It uses the `x-org` header to select an organization (falls back to session org, then first membership). It returns:

## Navigation Configuration

The sidebar navigation is fully dynamic â€” defined in the AI Governance workspace configuration and served via the IAM context endpoint.

### Menu Item Structure

Navigation is stored on the organization object and served within `org.navigation`. Menu items support groups with nested items, permission gating, and feature lists.

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier |
| `type` | string | `group` for collapsible category, omit for direct item |
| `label` | string | Display label |
| `icon` | string | Icon name (from curated icon map in the frontend) |
| `href` | string | URL path |
| `color` | string | Icon/label color |
| `target` | string | Link target (e.g., `_blank`) |
| `expanded` | boolean | Whether group starts expanded |
| `permissions` | string[] | Required permissions (item hidden if user lacks all of them) |
| `items` | array | Nested items (for groups) |
| `features` | array | Feature list entries `{ slug, agentId, label, permissions }` |
| `badge` | object | Badge configuration |
| `visibility.tiers` | string[] | Subscription tiers that can see this item |
| `visibility.features` | string[] | Feature flags required |

### Permission Filtering

The navigation is **server-side filtered** based on the user's role permissions before being returned. Items with `permissions` are checked via wildcard matching (e.g., `iam:*` matches `iam:members:manage`). Groups are only included if they have at least one visible sub-item after filtering. Feature lists within items are also filtered by permissions.

## Branding

Organizations can customize the platform appearance:

| Setting | Description |
|---------|-------------|
| `version` | Branding schema version |
| `platformName` | Custom platform name (replaces "Prisme.ai") |
| `description` | Platform description |
| `logo.light` | Logo URL for light theme |
| `logo.dark` | Logo URL for dark theme |
| `favicon` | Favicon URL |
| `colors.primary` | Primary brand color |
| `colors.secondary` | Secondary brand color |
| `colors.accent` | Accent color |
| `fonts.heading` | Heading font family |
| `fonts.body` | Body text font family |
| `customCssUrl` | URL to a custom CSS file for advanced theming |
| `customCssInline` | Inline CSS string |

Branding is applied by the `IAMProvider` in the platform shell. CSS variables are set on the root element, and the custom CSS file (if any) is injected as a `<link>` tag.

## Onboarding

When a user has no organization membership, the platform shows an **onboarding modal**:

1. **Join with invite code** â€” Enter a code shared by an organization admin
2. **Create organization** â€” Available only to users with `PlatformAdmin` role
3. **Auto-join** â€” If join rules match the user's email domain, they're automatically added

The onboarding flow is handled entirely by the frontend using the IAM context response.

## Organization Switching

```
POST v1/iam/switch-org
```

```json
{
  "orgSlug": "beta-inc"
}
```

Switches the user's active organization. Updates `user.meta.currentOrg`. The frontend reloads the IAM context to get the new organization's navigation, branding, and permissions.

The organizations list from the IAM context populates the org switcher dropdown in the sidebar.

## Degraded Mode

If no AI Governance workspace is configured (or it's unreachable), the platform shell falls back to **degraded mode**:

- Only the Builder link is shown in the sidebar
- No branding customization
- No organization context
- Basic user info from the OIDC session

This ensures the platform remains usable for development even without the governance layer.

## Service Account Context

```
GET v1/service-accounts/:id/context
```

Service accounts (used for machine-to-machine access) have their own context endpoint that returns permissions and scopes without navigation or branding.
