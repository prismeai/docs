AI Governance supports Single Sign-On (SSO) for organizations, allowing users to authenticate through their corporate identity provider.

## Endpoints

### Lookup SSO Configuration by Domain

```
GET v1/sso/config?domain=acme.com
```

Public endpoint (no auth required). Called by the Auth Service to determine the SSO provider for a user's email domain. Returns provider config **without secrets** (e.g., `clientSecret` is stripped).

### Get Full SSO Configuration

```
GET v1/sso/config-full
```

Returns the complete SSO configuration including secrets. Protected by internal auth token (`x-internal-auth` header), not user permissions — intended for the Auth Service only.

### Organization-Level SSO

```
PUT v1/orgs/:org_slug/sso
```

Self-service endpoint for organization owners to configure their SSO settings. Requires `iam:org:manage` permission.

## SSO Config Structure

SSO configuration is stored as part of the organization object (`ssoConfig` field):

```json
{
  "enabled": true,
  "domains": ["acme.com", "acme.co.uk"],
  "provider": {
    "type": "oidc",
    "oidc": {
      "issuer": "https://login.microsoftonline.com/{tenant}/v2.0",
      "clientId": "your-client-id",
      "clientSecret": "your-client-secret"
    }
  },
  "settings": {
    "enforceSSO": false,
    "autoProvision": true,
    "defaultRole": "user",
    "allowedRoles": ["user", "admin"]
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `enabled` | boolean | Whether SSO is active |
| `domains` | string[] | Email domains that trigger SSO (must be unique across orgs) |
| `provider.type` | string | `saml` or `oidc` |
| `provider.saml` | object | SAML config: `entryPoint`, `issuer`, `cert` |
| `provider.oidc` | object | OIDC config: `issuer`, `clientId`, `clientSecret` |
| `settings.enforceSSO` | boolean | Require SSO (disable local auth) |
| `settings.autoProvision` | boolean | Auto-create accounts for new SSO users |
| `settings.defaultRole` | string | Role assigned to auto-provisioned users (must exist in org) |
| `settings.allowedRoles` | string[] | Roles that SSO users can be assigned (stored but not currently enforced) |

    All three fields (`issuer`, `clientId`, `clientSecret`) are required. Works with Azure AD, Okta, Google Workspace, or any OIDC-compliant provider.

## Domain Validation

When updating SSO config, the system validates that domains are not already claimed by another organization. If a domain conflict is found, the update is rejected with a `DOMAIN_ALREADY_USED` error identifying the conflicting org.

## Integration with Platform Frontend

The platform frontend supports multiple auth providers via the `ENABLED_AUTH_PROVIDERS` environment variable:

```json
[
  { "name": "local" },
  { "name": "azure", "label": "Sign in with Azure" }
]
```

When SSO is configured, the sign-in page shows additional buttons for each provider alongside (or replacing) local authentication.

## Auto-Provisioning

When `autoProvision` is enabled, the SSO settings (stored here in AI Governance) are used by the Auth Service to provision accounts:
1. User authenticates via SSO for the first time
2. A platform account is automatically created by the Auth Service
3. The user is added to the organization with `defaultRole`
4. Join rules are also applied if configured

This eliminates the need for manual invitation for users authenticating through corporate SSO.
