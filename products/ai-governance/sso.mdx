AI Governance supports Single Sign-On (SSO) for organizations, allowing users to authenticate through their corporate identity provider.

## Endpoints

### Get SSO Configuration

```
GET v1/sso/config
```

Returns the public SSO configuration for the current organization (safe to expose to the frontend).

### Get Full SSO Configuration

```
GET v1/sso/config-full
```

Returns the complete SSO configuration including secrets. Requires admin permissions.

### Create / Update SSO Configuration

```
POST v1/sso/config
```

```json
{
  "provider": "azure",
  "config": {
    "client_id": "your-azure-client-id",
    "client_secret": "your-azure-client-secret",
    "tenant_id": "your-azure-tenant-id",
    "discovery_url": "https://login.microsoftonline.com/{tenant}/.well-known/openid-configuration"
  },
  "auto_provision": true,
  "default_role": "viewer"
}
```

### Organization-Level SSO

```
GET  v1/orgs/:org_slug/sso
POST v1/orgs/:org_slug/sso
```

Configure SSO at the organization level. Each organization can have its own identity provider.

## Configuration Options

| Field | Type | Description |
|-------|------|-------------|
| `provider` | string | Identity provider type (`azure`, `okta`, `google`, `saml`, `oidc`) |
| `config` | object | Provider-specific credentials and endpoints |
| `auto_provision` | boolean | Automatically create accounts for new SSO users |
| `default_role` | string | Role assigned to auto-provisioned users |

## Integration with Platform Frontend

The platform frontend supports multiple auth providers via the `ENABLED_AUTH_PROVIDERS` environment variable:

```json
[
  { "name": "local" },
  { "name": "azure", "label": "Sign in with Azure" }
]
```

When SSO is configured, the sign-in page shows additional buttons for each provider alongside (or replacing) local authentication.

## Auto-Provisioning

When `auto_provision` is enabled:
1. User authenticates via SSO for the first time
2. A platform account is automatically created
3. The user is added to the organization with `default_role`
4. Join rules (email domain matching) are also applied if configured

This eliminates the need for manual invitation for users authenticating through corporate SSO.
