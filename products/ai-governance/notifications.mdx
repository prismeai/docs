AI Governance provides two notification systems: **announcements** (platform-wide broadcasts) and **notifications** (per-user messages).

## Announcements

Announcements are platform-wide messages displayed to all users in an organization. They're used for product updates, maintenance notices, policy changes, etc.

### Endpoints

```
GET    v1/announcements                    # List announcements
POST   v1/announcements                    # Create announcement
GET    v1/announcements/:announcement_id   # Get announcement
PUT    v1/announcements/:announcement_id   # Update announcement
DELETE v1/announcements/:announcement_id   # Delete announcement
```

### Create Announcement

```json
{
  "title": "Platform Maintenance",
  "summary": "Scheduled maintenance on March 10, 2026",
  "content": "Full markdown content with tables, videos, etc.",
  "icon": "🔧",
  "link": "https://status.example.com",
  "orgSlug": "acme-corp",
  "targets": {
    "products": ["*"],
    "roles": ["admin", "owner"],
    "permissions": [],
    "users": []
  },
  "status": "active",
  "expires_at": "2026-03-10T04:00:00Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `title` | string | Announcement title |
| `summary` | string | Short description for card display |
| `content` | string | Full markdown content |
| `icon` | string | Emoji or image URL |
| `link` | string | Optional CTA link |
| `orgSlug` | string | Organization scope (`null` = platform-wide) |
| `targets` | object | Targeting rules (see below) |
| `status` | string | `draft`, `active`, `archived` |
| `expires_at` | datetime | Auto-archive date (purged 30 days after expiration) |

### Targeting Rules

Targeting combines **where** (products) and **who** (audience):

- `products` (required) — `["*"]` for all, or specific product slugs like `["secure-chat", "iam"]`. Uses OR logic (matches if the user is in any listed product).
- `roles`, `permissions`, `users` — Audience filters using OR logic between them. At least one audience field must be non-empty.

Logic: `product matches OR (role OR permission OR user matches)`

### Read Tracking

Per-user announcement read/dismiss state is tracked in the `user_announcement_state` collection. Each user has a `read` and `dismissed` flag per announcement.

## Notifications

Notifications are per-user messages generated by platform events.

### Endpoints

```
GET  v1/notifications                       # List user's notifications
GET  v1/notifications/:notification_id      # Get notification
POST v1/notifications/:notification_id/dismiss  # Dismiss notification
GET  v1/notifications/unread-count          # Get unread count
```

### Unread Count

```
GET v1/notifications/unread-count
```

Returns:
```json
{
  "count": 3
}
```

Used by the frontend to show a badge on the notification bell icon.

### Dismiss

```
POST v1/notifications/:notification_id/dismiss
```

Marks a notification as read/dismissed. Dismissed notifications no longer appear in the unread count.

## Frontend Integration

The **AI Engage** builtin app (`builtin-apps/ai-engage`) provides the frontend for managing announcements. It uses a Tiptap rich text editor for creating announcement content.

The platform shell displays notifications in the sidebar via the IAM context. The unread count is polled periodically and shown as a badge.
