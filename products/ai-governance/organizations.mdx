Organizations are the top-level tenant boundary in the platform. Every user belongs to one or more organizations, and all resources (agents, knowledge bases, API keys) are scoped to an organization.

## Organization Endpoints

### List Organizations

```
GET v1/orgs
```

Returns all organizations the authenticated user has access to. Platform admins see all organizations.

### Get Active Organization

```
GET v1/orgs/active
```

Lists active organizations (id, slug only). Designed for cross-workspace access by scheduled jobs.

### Create Organization

```
POST v1/orgs
```

```json
{
  "name": "Acme Corp",
  "slug": "acme-corp"
}
```

On creation, the platform automatically provisions:
- Default roles (`owner`, `admin`, `user`)
- A default subscription (free tier)
- The creating user as `owner`

### Get / Update Organization

```
GET v1/orgs/:org_slug
PUT v1/orgs/:org_slug
DELETE v1/orgs/:org_slug
```

Update supports changing `name`, `branding` (logo, colors, fonts, custom CSS), `navigation`, `joinRules`, and `settings` (including `settings.llm` for model governance, gated behind `llm:governance:manage` permission). Platform admins can also update `status` and `ssoConfig`.

## Members

### List Members

```
GET v1/orgs/:org_slug/members
```

Returns all members with their roles and group memberships.

### Update Member Role

```
PUT v1/orgs/:org_slug/members/:user_id
```

```json
{
  "roleSlug": "admin"
}
```

### Remove Member

```
DELETE v1/orgs/:org_slug/members/:user_id
```

### Invite Member

```
POST v1/orgs/:org_slug/members/invite
```

```json
{
  "email": "user@example.com",
  "roleSlug": "user"
}
```

Generates an invitation code that the user can redeem to join the organization.

## Invitations

### List Invitations

```
GET v1/orgs/:org_slug/invites
```

### Create Invitation

```
POST v1/orgs/:org_slug/invites
```

```json
{
  "code": "welcome-2024",
  "roleSlug": "user",
  "max_uses": 50,
  "expires_at": "2025-12-31T23:59:59Z"
}
```

### Delete Invitation

```
DELETE v1/orgs/:org_slug/invites/:invite_code
```

### Join Organization

```
POST v1/join
```

```json
{
  "invite_code": "welcome-2024"
}
```

Adds the authenticated user to the organization with the role specified in the invitation.

## Roles

Roles define what permissions a user has within an organization. There are built-in roles and custom roles.

### Built-in Roles

| Role | Permissions | Description |
|------|------------|-------------|
| `owner` | `*` (wildcard) | Full organization management |
| `admin` | `secure-chat:*`, `store:*`, `agent-builder:*`, `builder:*`, `knowledge:*`, `agents:use/manage/share`, `iam:members:manage`, `iam:groups:manage`, `iam:branding:manage`, `iam:navigation:manage`, `iam:join-rules:manage`, `iam:invites:manage`, `iam:subscriptions:manage`, `analytics:view` | Organization administration |
| `user` | `secure-chat:use`, `store:browse`, `agents:use`, `iam:groups:read` | Standard user access |

### Custom Roles

```
GET    v1/orgs/:org_slug/roles
POST   v1/orgs/:org_slug/roles
GET    v1/orgs/:org_slug/roles/:role_slug
PUT    v1/orgs/:org_slug/roles/:role_slug
DELETE v1/orgs/:org_slug/roles/:role_slug
```

Create roles with specific permissions:

```json
{
  "slug": "analyst",
  "name": "Data Analyst",
  "permissions": [
    "agents:read",
    "agents:use",
    "insights:read",
    "collections:read",
    "collections:write"
  ]
}
```

## Groups

Groups allow organizing users for bulk access control. Agents can be bound to groups (via [Agent Factory access control](/products/agent-factory/agents-api#access-control)), giving all group members access.

### CRUD

```
GET    v1/orgs/:org_slug/groups
POST   v1/orgs/:org_slug/groups
GET    v1/orgs/:org_slug/groups/:group_slug
PATCH  v1/orgs/:org_slug/groups/:group_slug
DELETE v1/orgs/:org_slug/groups/:group_slug
```

### Group Members

```
GET    v1/orgs/:org_slug/groups/:group_slug/members
POST   v1/orgs/:org_slug/groups/:group_slug/members
DELETE v1/orgs/:org_slug/groups/:group_slug/members/:user_id
```

When a group is deleted, the `iam.group.deleted` event is emitted, and Agent Factory automatically cleans up all agent bindings referencing that group.

## Join Rules

Join rules allow automatic organization membership based on generic field matching against the user object. This supports email domains, group membership, or any user attribute:

```json
{
  "field": "email",
  "matchValue": "@acme.com",
  "orgSlug": "acme-corp",
  "roleSlug": "user"
}
```

| Field | Description |
|-------|-------------|
| `field` | User object field path (e.g., `email`, `groups`, `authData.google.hd`) |
| `matchValue` | Value to match (uses `matches` operator for substring matching) |
| `orgSlug` | Organization to auto-join |
| `roleSlug` | Role to assign on auto-join |

Join rules are evaluated during `internal/resolve-context` (i.e., on the user's next IAM context call). When a match is found and the user isn't already a member, they are automatically added with the org's default subscription.

Changes to join rules trigger the `joinrules.sync` event for cache invalidation.

## Users

```
GET    v1/users
PATCH  v1/users/:user_id
DELETE v1/users/:user_id
```

User endpoints allow platform admins to list, update status (`pending`, `validated`, `deactivated`), or delete users. Deleting a user cascades membership and group cleanup across all organizations.

## IAM Membership

```
GET v1/iam/memberships
```

Returns all organization memberships for the authenticated user, enriched with org name, status, role name, and subscription ID.

```
POST v1/iam/switch-org
```

```json
{
  "orgSlug": "acme-corp"
}
```

Switches the user's active organization. Updates `user.meta.currentOrg`.
