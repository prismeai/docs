Organizations are the top-level tenant boundary in the platform. Every user belongs to one or more organizations, and all resources (agents, knowledge bases, API keys) are scoped to an organization.

## Organization Endpoints

### List Organizations

```
GET v1/orgs
```

Returns all organizations the authenticated user has access to. Platform admins see all organizations.

### Get Active Organization

```
GET v1/orgs/active
```

Lists active organizations (id, slug only). Designed for cross-workspace access by scheduled jobs.

### Create Organization

```
POST v1/orgs
```

```json
{
  "name": "Acme Corp",
  "slug": "acme-corp"
}
```

On creation, the platform automatically provisions:
- Default roles (`owner`, `admin`, `user`)
- A default subscription (free tier)
- The creating user as `owner`

### Get / Update Organization

```
GET v1/orgs/:org_slug
PUT v1/orgs/:org_slug
DELETE v1/orgs/:org_slug
```

Update supports changing `name`, `branding` (logo, colors, fonts, custom CSS), `navigation`, `joinRules`, and `settings` (including `settings.llm` for model governance, gated behind `llm:governance:manage` permission). Platform admins can also update `status` and `ssoConfig`.

## Members

### List Members

```
GET v1/orgs/:org_slug/members
```

Returns all members with their roles and group memberships.

### Update Member Role

```
PUT v1/orgs/:org_slug/members/:user_id
```

```json
{
  "roleSlug": "admin"
}
```

### Remove Member

```
DELETE v1/orgs/:org_slug/members/:user_id
```

### Invite Member

```
POST v1/orgs/:org_slug/members/invite
```

```json
{
  "email": "user@example.com",
  "roleSlug": "user"
}
```

Generates an invitation code that the user can redeem to join the organization.

```

```


```
POST v1/join
```

```json

  "invite_code": "welcome-2024"
}
```

Adds the authenticated user to the organization with the role specified in the invitation.

## Roles

Roles define what permissions and resource scopes a user has within an organization. There are system roles, environment-configured roles, and custom roles. System roles (except `owner`) can be overridden per organization.

### System Roles

| Role | Slug | Permissions | Scopes | Description |
|------|------|------------|--------|-------------|
| Owner | `org:owner` | `*` | `*` | Full organization management. Cannot be overridden. |
| Admin | `org:admin` | `orgs:members:manage`, `orgs:groups:manage`, `orgs:branding:manage`, `orgs:navigation:manage`, `orgs:invites:manage`, `orgs:join-rules:manage`, `orgs:apikeys:manage`, `users:manage`, `secure-chat:*`, `agent-factory:*`, `builder:*`, `engage:*`, `storage:*`, `collections:*`, `insights:*`, `ai-governance-v2:*` | `*` | Organization administration |
| Member | `org:member` | `orgs:roles:read`, `users:read`, `orgs:groups:read`, `orgs:members:read`, `agent-factory:agents:read`, `agent-factory:agents:explore`, `storage:vector_stores:read`, `storage:files:read`, `storage:skills:read`, `secure-chat:*` | — | Basic organization member |

Additional system roles can be defined via the `ORG_ADDITIONAL_SYSTEM_ROLES` environment variable (JSON array). Default additional roles when the env var is not set:

| Role | Slug | Permissions | Description |
|------|------|------------|-------------|
| Agent Maker | `agent-maker` | Member base + `agent-factory:*`, `storage:*`, `knowledge:*` | Full agent-factory & knowledge access |
| Builder | `builder` | Agent Maker + `builder:*` | Builder + agent-factory + knowledge access |
| Agent Standard | `agent-standard` | `llm:*`, `tools:*` | Default agent role (for service accounts) |

### Overriding System Roles Per Organization

All system roles except `owner` can be customized per organization. When an admin edits a system role (e.g. `org:admin`) in the AI Governance UI, a custom role override is saved for that organization. The custom override **shadows** the system definition — it fully replaces the system role's permissions and scopes for that org. The override is stored as a custom role with the same slug and is cached per-org in the API gateway.

### Custom Roles

```
GET    v1/orgs/:org_slug/roles
POST   v1/orgs/:org_slug/roles
GET    v1/orgs/:org_slug/roles/:role_slug
PUT    v1/orgs/:org_slug/roles/:role_slug
DELETE v1/orgs/:org_slug/roles/:role_slug
```

Create roles with specific permissions and optional resource scopes:

```json
{
  "slug": "analyst",
  "name": "Data Analyst",
  "permissions": [
    "agent-factory:agents:read",
    "agent-factory:agents:explore",
    "storage:vector_stores:read",
    "storage:files:read"
  ],
  "scopes": [
    "storage:vector_stores:kb_abc123",
    "agent-factory:agents:agent_xyz"
  ]
}
```

### Permission Format

Permissions follow a hierarchical `product:resource:action` pattern with wildcard support:

| Product | Key | Example Permissions |
|---------|-----|-------------------|
| Organization | `orgs` | `orgs:members:manage`, `orgs:groups:read`, `orgs:apikeys:create`, `orgs:branding:update`, `orgs:sso:manage`, `orgs:service-accounts:manage` |
| Users (Platform) | `users` | `users:read`, `users:update`, `users:delete`, `users:manage` |
| LLM | `llm` | `llm:completions:use`, `llm:embeddings:use`, `llm:governance:manage` |
| Tools | `tools` | `tools:web_search`, `tools:knowledge_search`, `tools:code_execution`, `tools:api_call` |
| Agents | `agent-factory` | `agent-factory:agents:read`, `agent-factory:agents:write`, `agent-factory:agents:delete`, `agent-factory:agents:publish`, `agent-factory:agents:share` |
| Storage / Knowledge | `storage` | `storage:vector_stores:read/write/share/delete`, `storage:files:read/write/delete`, `storage:skills:read/write/delete` |
| AI Governance | `ai-governance-v2` | `ai-governance-v2:observability:read`, `ai-governance-v2:observability_platform:read`, `ai-governance-v2:announcements:write` |

Wildcards are supported at any level: `orgs:members:*` grants all member actions, `storage:*` grants all storage permissions, `*` grants everything.

### Resource Scopes

Scopes restrict **which specific resources** a role can access, independent of permissions. They use the format `product:resource:id`:

```json
{
  "scopes": [
    "agent-factory:agents:agent_abc123",
    "storage:vector_stores:kb_xyz",
    "llm:models:gpt-4o"
  ]
}
```

- `*` grants access to all resources
- `agent-factory:agents:*` grants access to all agents
- If scopes are empty or omitted, no scope restriction is applied (all resources accessible)

Scopes are available on both roles and API keys. Some products don't support instance-level scoping (e.g. `orgs`, `ai-governance-v2`) — for these, the scope UI is hidden in the role editor.

### Hierarchical Permission Tree (UI)

The AI Governance role editor displays permissions as a hierarchical tree organized by product > resource > action. Key behaviors:

- Checking a product node sets the product-level wildcard (e.g. `storage:*`)
- Checking a resource node sets the resource-level wildcard (e.g. `storage:vector_stores:*`)
- Individual actions can be toggled independently
- Children are visually disabled when a parent wildcard is active
- A counter shows checked/total actions per resource and product
- For products supporting instance scoping, a "Scopes" section appears under each resource, allowing admins to set the "All resources" wildcard or add specific resource IDs
- Custom (non-tree-managed) permissions can be added manually via a free-text input

## Groups

Groups allow organizing users for bulk access control. Agents can be bound to groups (via [Agent Factory access control](/products/agent-factory/agents-api#access-control)), giving all group members access.

### CRUD

```
GET    v1/orgs/:org_slug/groups
POST   v1/orgs/:org_slug/groups
GET    v1/orgs/:org_slug/groups/:group_slug
PATCH  v1/orgs/:org_slug/groups/:group_slug
DELETE v1/orgs/:org_slug/groups/:group_slug
```

### Group Members

```
GET    v1/orgs/:org_slug/groups/:group_slug/members
POST   v1/orgs/:org_slug/groups/:group_slug/members
DELETE v1/orgs/:org_slug/groups/:group_slug/members/:user_id
```

When a group is deleted, the `iam.group.deleted` event is emitted, and Agent Factory automatically cleans up all agent bindings referencing that group.

## Join Rules

Join rules allow automatic organization membership based on generic field matching against the user object. This supports email domains, group membership, or any user attribute:

```json
{
  "field": "email",
  "matchValue": "@acme.com",
  "orgSlug": "acme-corp",
  "roleSlug": "user"
}
```

| Field | Description |
|-------|-------------|
| `field` | User object field path (e.g., `email`, `groups`, `authData.google.hd`) |
| `matchValue` | Value to match (uses `matches` operator for substring matching) |
| `orgSlug` | Organization to auto-join |
| `roleSlug` | Role to assign on auto-join |

Join rules are evaluated during `internal/resolve-context` (i.e., on the user's next IAM context call). When a match is found and the user isn't already a member, they are automatically added with the org's default subscription.

Changes to join rules trigger the `joinrules.sync` event for cache invalidation.

## Users

```
GET    v1/users
PATCH  v1/users/:user_id
DELETE v1/users/:user_id
```

User endpoints allow platform admins to list, update status (`pending`, `validated`, `deactivated`), or delete users. Deleting a user cascades membership and group cleanup across all organizations.

## IAM Membership

```
GET v1/iam/memberships
```

Returns all organization memberships for the authenticated user, enriched with org name, status, role name, and subscription ID.

```
POST v1/iam/switch-org
```

```json
{
  "orgSlug": "acme-corp"
}
```

Switches the user's active organization. Updates `user.meta.currentOrg`.
